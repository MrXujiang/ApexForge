AI 建模后端的服务化实现

后端承担生成任务编排、LLM 调用、Harness 多 Agent 输出、模板管理、资产记录、反馈收集和质量评分等职责。它把不稳定的自然语言输入转化为前端可以稳定消费的结构化建模参数。

后端关键文件

generation.service.ts generation-harness.service.ts llm.service.ts validation.service.ts templates.service.ts schema.prisma

API 模块地图

GenerationModule

核心生成模块，提供 /api/v1/generations 创建生成任务、查询历史和获取生成详情。

LlmModule

封装模型供应商配置、OpenAI-compatible 请求体、系统 Prompt、API Key 优先级和 JSON 解析。

ValidationModule

对生成结果进行基础质量判断，为前端展示质量报告和后续优化提供依据。

TemplatesModule

管理模板库和 seed 数据，为用户提供可快速开始的建模场景。

AssetsModule

记录模型资产与相关元数据，为导入、版本和资产库扩展打基础。

FeedbackModule

收集用户反馈与评分，形成模型质量迭代闭环。

后端请求链路

Controller

接收 DTO，统一 API 路由

GenerationService

推断品类并创建任务

LlmService

调用 DeepSeek / Kimi / Qwen

Harness

生成 Agent 与技能链说明

Prisma

保存任务、评分和资产

CreateGenerationDto

生成请求包含 prompt、category、provider 和 llmApiKeys。llmApiKeys 是在线模型配置能力的关键字段，允许前端在不修改 .env 的情况下传递当前浏览器的供应商 Key。

{
  "prompt": "生成一个复古便携式收音机",
  "category": "product",
  "provider": "deepseek",
  "llmApiKeys": {
    "deepseek": "在线填写的 Key",
    "kimi": "",
    "qwen": ""
  }
}

LLM 适配策略

统一供应商标识

normalizeProvider 将前端传入值归一化为 deepseek、kimi、qwen 等内部供应商标识。

解析 Provider Config

resolveProviderConfig 根据供应商获取 baseURL、模型名、环境变量名和展示名称。

API Key 优先级

先使用 apiKeyOverride，也就是用户在线填写的 Key；如果没有，才读取服务端环境变量。

OpenAI-compatible 请求

后端向 /chat/completions 发送 messages、temperature 和 response_format json_object，要求模型返回结构化 JSON。

错误信息产品化

缺少 Key 时给出可操作提示：在左侧在线配置填写，或在项目环境变量中设置对应变量名。

Prisma 数据模型

模型用途扩展价值
Template保存模板名称、品类、提示词和默认参数支持模板市场和官方示例库
GenerationTask记录一次生成任务的 prompt、category、status、result支持历史、复现、任务队列
ValidationReport保存生成结果的验证说明支持质量审计和自动修复建议
QualityScore记录几何、材质、提示词匹配等评分支持排序、推荐和 A/B 测试
ModelAsset记录模型资产信息支持资产库、下载、版本管理
ModelVersion保存资产不同版本支持迭代对比和回滚
Feedback保存用户反馈支持产品闭环和训练数据沉淀
后端开发建议：

新增模型供应商时优先扩展 provider config 和 normalizeProvider；新增生成参数时同步更新 DTO、共享类型、LLM 系统 Prompt、前端属性面板和 modelFactory 消费逻辑。
